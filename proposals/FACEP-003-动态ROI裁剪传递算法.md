# 提案：动态 ROI 裁剪 — 算法输入仅传人脸采样区域

> 提案编号：FACEP-003  
> 创建日期：2026-07-01  
> 状态：已实现

---

## 1. 动机

当前算法处理全图 1600×1300（~4MB），DSP 推理 ~70ms。实际有用的只有人脸周围区域，大量像素浪费在无关背景上。通过 ROI 裁剪可以：

- 减小算法输入数据量（4MB → 约 0.8MB），降低 memcpy 和 DSP 带宽
- 为未来提高算法帧率留出空间

## 2. 裁剪窗口规格

### 2.1 尺寸

**固定正方形**。

```
winSize = 900px
```

裁剪窗口恒为 **900×900**，在原图坐标系下。

### 2.2 位置——跟随人脸

裁剪窗口跟随人脸位置，目标是让人脸处于窗口的黄金分割点（0.382）：

```
┌─────────────────────┐
│     H*0.382         │
│      ┌─────────┐    │
│      │  ● 人脸  │    │  ← 人脸中心在黄金分割点
│      │  (虚线)  │    │
│      │  H*0.618  │    │
│      │          │    │
│      └─────────┘    │
│                     │
└─────────────────────┘
```

**跟踪规则：**

```
winCenterY = faceCenterY + winSize * 0.118
winCenterX = faceCenterX
```

即人脸中心在窗口纵向 0.382 处（黄金分割点）。

**边缘约束：**

当窗口触及图像边界时，钳位到边缘，不再移动：

```
winLeft = clamp(winCenterX - winSize/2, 0, imgW - winSize)
winTop  = clamp(winCenterY - winSize/2, 0, imgH - winSize)
```

此时人脸位置继续变化，但窗口位置固定，人脸在窗口内移动。

### 2.3 无人脸时的行为

- 首帧或人脸丢失：窗口居中
- 后续检测到人脸：按跟踪规则移动
- 连续丢失人脸超过阈值（如 30 帧）：回到居中

## 3. 数据流

```
修改前：
  全图 byte[] (1600×1300, 4MB) → processFrame → DSP ~70ms

修改后：
  全图 byte[] (1600×1300)
    │
    ├── EvsGL20CameraRenderer 渲染全图（不变）
    │
    └── 裁剪 ROI (900×900, ~1.6MB) → processFrame → DSP 减少
```

裁剪在算法线程的 `processFrame` 之前完成，从已读取的 `byte[]` 中截取 ROI 区域。

## 4. UI 绘制

在 `FaceOverlayView` 上绘制黄色虚线矩形框表示当前裁剪区域：

- 颜色：黄色（`Color.YELLOW`）
- 样式：虚线描边（`DashPathEffect([10, 8], 0)`），线宽 2px
- 位置：跟随裁剪窗口坐标（与原图坐标一致，`setFaces` 时随 imgW/imgH 缩放）
- 无需标签文字

## 5. 修改范围

| 文件 | 修改 | 说明 |
|---|---|---|

| `FrameProcessor.kt` | 传递人脸框信息到裁剪逻辑 | 接口调整 |

## 6. 效果预期

- **算法输入**：全图 4MB（1600×1300 UYVY）→ 裁剪 1.6MB（900×900 UYVY），减少 ~60%
- **DSP 推理耗时**：~70ms → ~20-30ms（预估，取决于模型输入尺寸适配）
- **全图显示**：不变，用户看到的是完整画面
- **裁剪窗口**：实时跟随人脸，边缘自适应钳位

## 7. 风险

- 模型输入尺寸：确认 DSP 模型是否接受可变输入尺寸。若不可变，则需要补边（padding）到模型要求的固定尺寸
- 人脸快速移动时窗口可能有延迟：需评估跟踪平滑度，必要时加插值
- 人脸贴近边缘时裁剪区域可能只包含半张脸，影响识别精度
